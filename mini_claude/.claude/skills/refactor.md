# /refactor — 代码重构

请执行以下步骤：

## 1. 分析目标
- 如果有参数（文件路径），分析指定文件
- 否则询问用户要重构哪部分代码

## 2. 识别问题
逐一列出代码中的坏味道：
- **过长函数**: > 50 行的函数，考虑拆分
- **重复代码**: 相似逻辑出现 3 次以上，考虑抽取
- **深层嵌套**: > 3 层的 if/for 嵌套，考虑提前返回
- **命名问题**: 单字母变量、含义不清的名称
- **耦合过紧**: 一个改动牵连多处，考虑解耦
- **过大的类**: 职责不单一，考虑拆分

## 3. 制定方案
给出重构计划：
```
目标：描述重构要达到的效果
方案：
1. 具体步骤 1
2. 具体步骤 2
...
影响范围：列出会修改的文件
风险评估：低/中/高
```

## 4. 执行重构
- 向用户确认方案后再动手
- 每个步骤做完后运行测试确保不破坏功能
- 保持行为不变，只改结构

原则：
- 不要过度抽象，三次以下重复不用提取
- 不要改变公开 API 的签名
- 重构和功能修改分开提交
