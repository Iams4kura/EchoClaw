# /review — 代码审查

请执行以下步骤：

1. 确定审查范围：
   - 如果有参数（文件路径），审查指定文件
   - 否则运行 `git diff` 审查当前未提交的变更
   - 如果没有变更，运行 `git diff HEAD~1` 审查最近一次提交
2. 逐文件分析，关注以下维度：
   - **安全性**: SQL 注入、XSS、命令注入、硬编码密钥、不安全的反序列化
   - **Bug 风险**: 空指针、数组越界、竞态条件、资源泄漏、类型错误
   - **逻辑问题**: 边界条件、错误处理缺失、逻辑矛盾
   - **性能**: N+1 查询、不必要的循环、大内存分配
   - **可维护性**: 过长函数、重复代码、命名不清晰

3. 输出格式：
   ```
   ## 📋 审查摘要
   变更范围：X 个文件, +Y/-Z 行

   ## 🔴 必须修复 (Critical)
   - file.py:42 — 描述问题和修复建议

   ## 🟡 建议改进 (Warning)
   - file.py:18 — 描述问题和改进方向

   ## 🟢 整体评价
   一句话总结代码质量
   ```

4. 如果代码没什么问题，直接说 "LGTM" 不要硬凑问题
